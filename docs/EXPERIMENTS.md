# Журнал спроб

Короткий опис того, що я пробував, які результати отримав і які напрями відкинув. Повний трекінг
(метрики, гіперпараметри, артефакти, графіки по кожному прогону) — у публічному W&B, перегляд доступний
без авторизації:

https://wandb.ai/denkondrat13-set-university/phishing-scorer

Зазначу одразу: ваги в здачу не входять. Числа рахуються на локальних вагах (`max_length=256`, MPS), а
W&B я використовую як трекер експериментів, а не як сховище ваг. Для повторного прогону ноутбук
самодостатній (F0 будує дані з HF, F2/F9 тренують, F7 експортує ONNX).

## Порівнювані прогони (W&B)

| Run | F1 (public test) | Конфіг | Опис |
|---|---|---|---|
| `baseline-tfidf-logreg` | 0.981 | TF-IDF + LogReg | точка відліку |
| `distilbert-base` | 0.976 | 2 еп, batch 16, lr 2e-5, max_len **20** | коротко-контекстний експеримент (див. нижче) |
| `distilbert-aug` | 0.972 | 2 еп, batch 16, lr 1e-5, max_len **20** | дотренована на корпоративному синтетичному міксі |
| капстоун (локально) | 0.976 / aug 0.973 | 2 еп, batch 16, lr 2e-5, max_len **256** | канонічні числа ноутбука |

## Абляція: довжина контексту (20 vs 256)

Це навмисний експеримент і, на мою думку, найінформативніший результат. Я натренував DistilBERT із
`max_length=20`, тобто модель бачить лише перші ~20 токенів листа. Очікував помітного падіння якості, але
F1 майже не змінився (0.976 / 0.972 проти 0.976 / 0.973 на 256 токенах).

Висновок: на публічному тесті рішення визначається кількома токенами. Це майже пряме експериментальне
підтвердження leakage — модель спирається на назви корпусів і спам-маркери, а не на глибоку семантику.
Результат підсилює всю аргументацію: саме тому й був потрібен окремий корпоративний бенчмарк, а не довіра
до 0.98.

Зауваження: моделі в W&B (max_len 20) — це інша конфігурація, ніж локальні ваги ноутбука (256). Значення
F1 близькі не через ідентичність моделей, а через лексичну простоту задачі.

## Середовища тренування

| Середовище | Роль | Параметри |
|---|---|---|
| Локально (Intel Mac, MPS) | baseline + канонічні прогони капстоуна + оцінювання в ноутбуці | batch 16 @ 256, 2 епохи |
| Colab GPU | важчі експерименти DistilBERT (у т.ч. абляція max_len 20), логи в W&B | GPU дозволяє більший batch / більше епох |

batch 32 @ 256 на MPS призводить до OOM (~6.77 GB), тож я лишив 16. fp16 на MPS не підтримується.

## Відкинуті напрями

| Спроба | Причина відмови |
|---|---|
| DeBERTa-v3 | ламається ONNX-експорт, а потрібен був чистий CPU-serving |
| ModernBERT | завелика для скорингу на кожне натискання клавіші на CPU |
| `optimum` для ONNX | несумісний з transformers 4.57, тож експорт робив вручну |
| batch 32 @ 256 на MPS | OOM, повернувся до batch 16 |
| fp16 на MPS | не підтримується |
| інший публічний «OOD» набір | 81% збіг із train, тобто це не справжній OOD; замінив власним корпоративним бенчмарком |

## Ключові числа (канонічні, max_len 256)

- Калібрація (temperature scaling): T=1.33, ECE 0.0106→0.0052, Brier 0.0153→0.0148.
- ONNX-INT8: 268→67 MB (×4), `max|Δp_phish|`=0.0014, F1 без втрат.
- Латентність serving (заміряно): p50 10.5 / p95 12.3 мс на CPU.
- Augmentation проти domain gap: recall очевидних приманок 0.25→0.85, без втрат на public.
- Spearman шкали проти ручної розмітки: ρ=0.486 (тільки фішинг) / 0.792 (Safe→0).
- Порівняння з LLM (zero-shot Opus 4.8, n=60): F1 1.00, нарівні з моєю моделлю; різниця в латентності та вартості.
