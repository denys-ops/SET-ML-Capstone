## 0. TL;DR

Тренуємо легкий калібрований класифікатор `P(phish)` на публічних email-датасетах і доводимо
тезу дистиляції: **мала модель тримає ~якість LLM при у ~сотні разів нижчій латентності**.
Результат капстоуну — **один наскрізний ноутбук** (Colab), яким можна пройтися й розповісти кожен
крок, плюс **збережений модель-артефакт** (ваги + калібратор + препроцесинг), який окремо
вбудовується у Whalen як реактивна шкала «очевидності фішингу».

---

## 1. Цілі та не-цілі

### Цілі (збалансовано три ролі)
1. **Академічна повнота** — повний DL-цикл із чесними метриками й висновками (оцінка за капстоун).
2. **Портфоліо/співбесіда** — чистий наратив дистиляції + таблиця-висновок «своя модель vs zero-shot LLM»
   (якість × латентність × вартість).
3. **Реальний артефакт** — натренована каліброана модель, придатна для вбудови у Whalen-фічу.

### Не-цілі (свідомо поза скоупом цього капстоуну)
- ❌ Продакшн-сервінг: ONNX-runtime в інфрі, FastAPI-ендпоінт, FE-шкала в редакторі — **окрема Whalen-робота**.
- ❌ Тренування на приватних Whalen-даних (`was_clicked` тощо) — приватні, розріджені, лише як концептуальна валідація.
- ❌ Перемога над GPT в якості. Ціль — trade-off (якість ≈ при латентності/вартості ≪).

### Межа handoff
Капстоун **закінчується** на: збережений артефакт + бенчмарк-таблиця + (опційно) ONNX-експорт як
формат передачі. Усе після цього (інтеграція ендпоінта, дебаунс, перефарбування) — у репо Whalen.

## 4. Архітектура рішення (концептуально)

```
text (subject + body, HTML-stripped, {{.URL}}-normalized)
        │
        ▼
   ┌──────────────┐      ┌────────────────────┐      ┌──────────────────┐
   │ DistilBERT   │ ──►  │ logit → sigmoid     │ ──►  │ Calibrator (T)   │ ──►  P(phish) ∈ [0,1]
   │ fine-tuned   │      │  P_raw(phish)       │      │ temperature scale│
   └──────────────┘      └────────────────────┘      └──────────────────┘
                                                              │
                          color = invert(P)  ◄────────────────┤  (low P = переконливо, high P = очевидно)
                          spans = token attribution ◄─────────┘
```

**Чому % первинний:** бінарний класифікатор усередині рахує ймовірність (`logit→sigmoid→P`), а
клас — це вже поріг. Для шкали просто **не порогуємо**. Колір = інверсія `P(phish)`.

## 5. План даних

### 5.1 Джерела (лише публічні — тренування)
| Джерело | Клас | Роль |
|---|---|---|
| Nazario Phishing Corpus | phish (1) | реальні фішинг-листи |
| SpamAssassin (ham) / Enron | legit (0) | легітимна пошта |
| Kaggle «Phishing Email Detection» (~18k) | змішаний | альтернатива/доповнення, крос-перевірка |
| Whalen 5 platform-шаблонів | — | **inference/demo only**, НЕ train |

### 5.2 Цільова схема
Уніфікувати все до `text, label`, де `text = subject + "\n" + body_plaintext`.

### 5.3 Препроцесинг (критично — впливає на чесність)
1. **HTML-strip** → плейн-текст (зберегти subject окремо, потім зклеїти).
2. **Нормалізація `{{.URL}}`** та інших Gophish-плейсхолдерів → не давати моделі «дивний лінк» як фічу
   (інакше вона вчиться на артефакт, а не на соц-інженерію).
3. Нормалізація URL/email/чисел до плейсхолдерів (`<URL>`, `<EMAIL>`, `<NUM>`) — щоб модель не
   запам'ятовувала конкретні домени з тренувального корпусу.
4. Видалення службових заголовків листів (Received:, X-*) — це лік-сигнали джерела, не фішингу.
5. Дедуп (точний + near-dup за хешем шинглів) **до** спліту.

### 5.4 Лік-чеки (обов'язково — типова пастка цього домену)
- **Source leakage**: якщо весь phish з Nazario, а весь legit з Enron, модель вивчить «стиль корпусу»,
  а не фішинг. Перевірити: чи можна вгадати label лише за службовими токенами? → почистити заголовки,
  розглянути змішування джерел.
- **Дублі між train/test** після мерджу датасетів → дедуп до спліту.
- **Часовий/доменний лік** → за можливості тримати домени/відправників лише в одному спліті.

### 5.5 Спліт
- Stratified train / val / test (напр. 70/15/15), фіксований seed.
- **val** — для тюнінгу та фіту калібратора (або окремий calib-split, щоб не калібрувати на train).
- **test** — лише фінальний eval і **спільний** з LLM-бенчмарком.

## 6. План моделювання

| Етап | Метод | DL-субстанція / нотатки |
|---|---|---|
| **Baseline** | TF-IDF + LogReg (`predict_proba`) | чесна точка відліку; швидкий, інтерпретований |
| **Baseline+** (опц.) | TF-IDF + LightGBM | сильніший «не-DL» baseline для контрасту |
| **DL-модель** | fine-tune DistilBERT / MiniLM | трансформери, fine-tuning; HF `Trainer` або кастомний loop |
| **Калібрація** | Temperature scaling (дефолт) | ділить логіти на вивчену `T`; не міняє точність |
| | Platt / Isotonic (`CalibratedClassifierCV`) | для порівняння; isotonic потребує більше даних |
| **Explainability** | Integrated Gradients (Captum) або SHAP | token attribution → spans для підсвітки |
| **Стиснення** (stretch) | int8-quantize → ONNX export | latency-бюджет, бенчмарк розмір/швидкість |

**Налаштування тренування (стартові):** max_len ~256–512, lr 2e-5, batch 16–32, 3–4 епохи,
early stopping за val F1, weight decay 0.01, fp16 на GPU. Class imbalance → зважений loss або
зважений семплінг (рішення після EDA §5).

## 7. План оцінки (головний висновок капстоуну)

### 7.1 Метрики
- **Класифікація**: F1, Precision/Recall, ROC-AUC, PR-AUC, confusion matrix.
- **Калібрація**: ECE + reliability diagram (до/після).
- **Сервінг-проксі**: latency p50/p95 на CPU, розмір моделі (для наративу, навіть без сервінгу).

### 7.2 Бенчмарк vs zero-shot LLM (ядро портфоліо-наративу)
На **тому ж test-set**:
1. Прогнати zero-shot Claude/GPT з фіксованим промптом (phish? + впевненість).
2. Порівняти: якість (F1/AUC) × латентність (мс/виклик) × вартість ($/1k).
3. Таблиця-висновок: «своя модель = X% якості LLM при ~1/N латентності та ~$0 інференсу».

> Якщо доступ до LLM-API обмежений — зробити на підвибірці test (напр. 300–500 листів) і чесно
> це зазначити.

### 7.3 Валідація на Whalen-шаблонах (якісно)
Прогнати 5 platform-шаблонів через модель, показати score + spans. Очікування (записати як гіпотезу):
реалістичні приманки → нижчий `P(phish)` (🟢). Описати domain-gap-поведінку (§8.1), не претендувати на ground-truth.

## 8. Ризики та обмеження (в README/ноутбук)

1. **Domain gap**: публічні датасети ≠ навмисно реалістичні Whalen-приманки. Модель може бачити вдалу
   приманку як legit — це й є «зелений», але треба валідувати на 5 шаблонах і описати.
2. **Source leakage** (§5.4): найбільший ризик чесності метрик. Мітигуємо чисткою заголовків +
   дедупом + (за можливості) змішуванням джерел.
3. **Частковий текст** (для майбутньої фічі, не для тренування): score стрибатиме на недописаному
   листі → у фічі оцінювати лише при паузі і якщо тексту > N символів. У капстоуні — лише згадка.
4. **`{{.URL}}`-токен**: нормалізувати в препроцесингу, не вважати фічею.
5. **Over-claim калібрації**: каліброване `P(phish)` ≈ «наскільки читається як підручниковий фішинг» —
   це **проксі**, а не ground-truth susceptibility (її видно лише з `was_clicked`).

## 9. Handoff у Whalen-фічу (межа капстоуну)

Капстоун віддає **контракт артефакту**, фіча його споживає окремо:

- **Артефакт**: ваги DistilBERT + калібратор (`T`) + конфіг токенайзера/препроцесингу + threshold.
- **Логічний інтерфейс** (реалізує вже Whalen-робота): `text → { score: P(phish), label, color: invert(score), spans[] }`.
- **Рекомендований формат передачі**: ONNX (stretch §6) — щоб Whalen-сервінг не тягнув PyTorch.
- **Поза цим документом**: FastAPI-ендпоінт `POST /phishing-draft/score`, дебаунс ~250 мс на FE,
  quality-gate для drafter-output. Усе — в репо Whalen.

> Академічний код капстоуну живе окремо. У Whalen заходить лише натренована модель + тонкий шар.

## 14. Відкриті питання (вирішити по ходу)

- Чи змішувати джерела всередині кожного класу для боротьби з source leakage, чи залишити «as-is»
  і просто чесно описати ризик? (рішення після EDA §5.4)
- Який саме LLM для бенчмарку (Claude / GPT) і чи на повному test, чи на підвибірці (залежить від доступу/$).
- Чи додавати другу структурну HTML-голову (LightGBM на handcrafted features) — наразі поза основним
  скоупом, тримаємо як можливе розширення.
