# Phishing-Obviousness Scorer

Компактна калібрована модель, яка за текстом листа повертає `P(phish)` — оцінку того, наскільки лист
схожий на фішинг. Я трактую цю ймовірність як шкалу «очевидності»: низька оцінка означає, що приманка
переконлива, висока — що фішинг легко розпізнати. Додатково модель підсвічує токени, які найбільше
впливають на рішення.

Мотивація: у редакторі навчальних фішинг-шаблонів корисно показувати автору, наскільки очевидний його
лист, безпосередньо під час набору. Викликати LLM на кожне натискання клавіші неможливо — це надто
повільно й дорого. Тому потрібна невелика швидка модель, здатна працювати на CPU, ще й з поясненнями.

Уточнення щодо семантики: для самого продукту вона інвертована (вдала навчальна приманка відповідає низькій
оцінці), але модель — звичайний бінарний класифікатор phish/legit, а «очевидність» — це не-порогований
`P(phish)`.

Склад репозиторію: дослідницький ноутбук `notebooks/phishing_scorer.ipynb` (F0–F10, читається зверху
вниз), FastAPI-сервінг у `serve/` та цей README. Повний MLOps-цикл (DVC, W&B, моніторинг, CI) винесено в
окремий проєкт (посилання нижче).

## Дані

**Що було потрібно.** Розмічений корпус email-текстів (phish/legit), публічний і відтворюваний, щоб
будь-хто міг його завантажити, і достатнього обсягу для fine-tune трансформера.

**Що я розглядав і що обрав.**
- **Основний — `zefang-liu/phishing-email-dataset`** (HuggingFace, ~18.6k листів, бінарний Safe /
  Phishing). Я обрав його, оскільки він публічний, завантажується напряму через `load_dataset` без
  авторизації, має чисту бінарну розмітку й достатній обсяг. Це агрегований набір реальних фішинг- та
  легітимних листів.
- **MeAJOR** (датасет із препринту) — розглядав, але він недоступний для прямого завантаження, тож заради
  відтворюваності відкинув.
- **Ще один публічний фішинг-набір** — розглядав як OOD-тест, але він на ~81% перетинається з тренувальним,
  тобто справжнім out-of-distribution не є; для цієї ролі не підійшов.

**Що довелося зробити самому.** Публічних даних для головної мети (корпоративні приманки) не вистачає, тож
я додатково підготував: held-out бенчмарк `data/synthetic_benchmark.csv` (60 корпоративних листів — 20
очевидних приманок / 20 тонких / 20 легіт, лише для оцінювання), генератор корпоративних приманок
`src/synth.py::gen()` для augmentation (стиль навмисно відрізняється від бенчмарку), і ручну розмітку
`data/labeled/labeled_v{1,2}.parquet` (Label Studio, вердикт + `obviousness` 0–10) для перевірки шкали.

**Підготовка.** Після препроцесингу (`html_to_scoring_text`) я прибрав порожні рядки й точні дублікати
**до** спліту (щоб уникнути витоку між train і test). Лишилося 17 171 листів (10 698 legit / 6 473 phish)
→ train 12 019 / val 2 576 / test 2 576, частка фішингу 0.377.

### Проблеми, які виявилися

1. **Source leakage.** Legit-листи походять з корпусу Enron і академічних розсилок, phish — зі
   спам-корпусів. Через це найінформативнішими ознаками виявляються назви корпусів (`enron`, `vince`,
   `university`), а не ознаки соціальної інженерії. Отже, публічні ~0.98 значною мірою є артефактом
   датасету. Пряме підтвердження — абляція `max_length` 20 vs 256: F1 майже не змінився (деталі у
   [`docs/EXPERIMENTS.md`](docs/EXPERIMENTS.md)).
2. **Немає справжнього OOD** серед доступних публічних наборів (той інший набір на ~81% збігається з
   тренувальним), тому чесну перевірку довелося будувати самому — корпоративний бенчмарк.
3. **Domain gap.** На корпоративних приманках публічна модель провалюється (recall очевидних 0.25) — це
   головна змістовна знахідка, до якої повертаюся нижче.

## Метод

Найважливіша спільна деталь — препроцесинг. Функція `src/preprocess.py::html_to_scoring_text` однаково
обробляє і тренувальні дані, і вхід у проді, щоб уникнути train/serve skew. Вона видаляє
`<head>/<style>/<script>`, бере видимий текст, додає subject спереду, розгортає Gophish-плейсхолдери й
нормалізує URL/EMAIL/NUM. Щоб дві копії функції (у `src/` та `serve/`) не розійшлися, є перевірка
`scripts/check_preprocess_sync.py`.

Модель — `distilbert-base-uncased`, fine-tune (2 логіти, `max_len=256`, 2 епохи, lr 2e-5, batch 16,
wd 0.01). Є дві версії: `distilbert` (базова, на публічних даних) і `distilbert_aug` (дотренована на
корпоративному синтетичному стилі; саме її я використовую за замовчуванням для корпоративних приманок).
Точкою відліку слугує TF-IDF + LogReg.

## Результати

Класифікація на held-out test (2 576 листів):

| Модель | F1 | ROC-AUC | PR-AUC |
|---|---|---|---|
| TF-IDF + LogReg (baseline) | 0.981 | 0.998 | 0.997 |
| DistilBERT (max_len 256, 2 епохи) | 0.976 | 0.998 | 0.997 |

Baseline і DistilBERT практично нарівні, що очікувано з огляду на лексичну простоту задачі (той самий
leakage). Отже, цінність DL тут не в сирій точності (її забезпечує й LogReg), а в калібрації, поясненнях
і можливості адаптувати модель під новий стиль (augmentation нижче).

## Калібрація

Temperature scaling дав `T = 1.33` (`data/calibration.json`). Скоринг: `P = softmax(logits/T)[:,1]`.

| | ECE | AECE | Brier |
|---|---|---|---|
| до (T=1) | 0.0106 | 0.0100 | 0.0153 |
| після (T=1.33) | 0.0052 | 0.0071 | 0.0148 |

ECE зменшилося вдвічі. Для шкали «очевидності» це важливіше за F1, оскільки я використовую саму
ймовірність, а не лише поріг. У ноутбуці наведено reliability diagram до/після.

## Пояснення (explainability)

Captum `LayerIntegratedGradients` (baseline `[PAD]`, n_steps=50, на CPU) дає внесок кожного токена. У
сервінгу це ендпоінт `/explain`, який лениво завантажує PyTorch, бо ONNX не надає градієнтів. Знак `+`
зсуває оцінку до phishing, `−` — до legit.

## ONNX та квантизація

Експорт виконано через `torch.onnx.export` + `onnxruntime.quantization.quantize_dynamic`. Пакет `optimum`
я не використовував, оскільки він несумісний з transformers 4.57, тож експорт робив вручну. Dynamic INT8:

| | Розмір | F1 |
|---|---|---|
| FP32 (PyTorch) | 268 MB | 0.976 |
| ONNX-INT8 | 67 MB (×4 менше) | ~0.977 (без втрат) |

Щоб «без втрат» не було лише декларацією: `max|Δ p_phish|` між FP32 та INT8 = 0.0014. Тобто квантизація
практично не вплинула на вихід.

## Serving

FastAPI у `serve/`. Основний бекенд `/predict` — ONNX-INT8 (швидкий), `/explain` — PyTorch+Captum на
вимогу. Контракт: `POST /predict {html, subject?} → {label, p_phish, raw_p, obviousness_1_10}`. Ендпоінти:
`/predict`, `/predict/batch`, `/explain`, `/health`, `/metrics` (Prometheus), `/docs` (Swagger).

Латентність `/predict` я заміряв на практиці (ONNX-INT8, CPU, по одному листу через HTTP, n=200):

| p50 | p95 | p99 | mean |
|---|---|---|---|
| 10.5 ms | 12.3 ms | 13.1 ms | 10.6 ms |

10 мс на CPU — це саме той результат, заради якого робилася вся робота: такий скоринг можна виконувати на
кожне натискання клавіші, а виклик LLM — ні. Латентність визначається архітектурою DistilBERT, тож для
базової та aug версій вона однакова.

Запуск локально:
```bash
pip install -r serve/requirements.txt
python scripts/fetch_models.py --local     # ваги локально (або з W&B, див. нижче)
# зібрати ONNX_DIR (токенайзер + model.int8.onnx) і стартувати:
BACKEND=onnx ONNX_DIR=<dir> MODEL_DIR=model/distilbert_aug uvicorn serve.main:app --port 8080
python serve/test_client.py                # демонстраційний прогін
```

## Чи відповідає шкала людському сприйняттю (Spearman)

Це моє головне обмеження: каліброване `P(phish)` — проксі, а не прямий вимір того, наскільки лист зловить
людину. Тому я взяв ручну розмітку, прогнав її через `html_to_scoring_text` + `distilbert_aug` (T=1.33) і
порахував рангову кореляцію модельної оцінки з людською `obviousness` (`src/eval_scale.py`, клітинка F10):

| Аналіз | n | Spearman ρ | p |
|---|---|---|---|
| Тільки фішинг (obviousness 1–10) | 27 | 0.486 | 0.010 |
| Повністю, Safe→0 | 45 | 0.792 | 9.6e-11 |

Бінарне розділення чисте (середнє `p_phish`: фішинг 0.591 vs Safe 0.001). Серед самих приманок кореляція
помірна й значуща — модель загалом ставить очевидніші приманки вище, але це справді проксі, а не точний вимір.
Обмеження: невеликий n і єдине джерело розмітки. Scatter — `docs/spearman_scale.png`.

## Головна знахідка — domain gap

Найцікавіший результат отримано тут. Прогін корпоративного бенчмарку (60 листів) через публічну модель:

| | recall очевидні | recall тонкі | legit spec. | bench-F1 | public F1 |
|---|---|---|---|---|---|
| публічна модель | 0.25 | 0.00 | 1.00 | 0.22 | 0.973 |
| після augmentation | 0.85 | 0.10 | 1.00 | 0.64 | 0.973 |

Публічна модель фактично вивчила відповідність «спам-словник = фішинг» і майже не розпізнає корпоративні
приманки (recall 0.25). Дотренування на ~600 синтетичних корпоративних листах підняло recall очевидних приманок
з 0.25 до 0.85, і при цьому не погіршило публічний тест та не додало нових false-positives. Тонкі приманки
(~0.10) лишаються відкритою проблемою — це чесна межа поточного підходу.

## Порівняння з LLM

Щоб перевірити тезу «а чи не простіше було одразу взяти LLM», я зробив реальний прогін, а не покладався на
числа з літератури. Взяв 60 листів із public test (30/30, seed 42), приховав мітки, і Claude Opus 4.8
класифікував їх zero-shot лише за текстом. Далі окремий скрипт звірив вердикти з істиною (артефакти в
`data/llm_eval/`).

| Модель | F1 | Acc | Латентність | Вартість |
|---|---|---|---|---|
| Claude Opus 4.8 (zero-shot) | 1.00 | 1.00 | ~сотні мс–секунди (API) | оплата за токени |
| DistilBERT-INT8 (моя) | 1.00 | 1.00 | 10.5 мс (CPU) | ~$0 |
| TF-IDF+LogReg (baseline) | 1.00 | 1.00 | <1 мс | ~$0 |

Усі три моделі дають ідеальний результат, 0 розбіжностей. Це не слабкий тест, а суть висновку: на public
data якість не є розрізняльним фактором (той самий leakage), розрізняють латентність і вартість. Моя
модель дає ту саму якість у сотні разів швидше й за майже нульову вартість, що й уможливлює реактивний
скоринг на кожне натискання; для LLM це недосяжно. Реальне розходження в якості проявляється на
корпоративному domain gap (вище), а не на public test.

Застереження щодо методу: одна LLM-сесія, n=60, лише текст. Латентність і вартість LLM — порядкові оцінки,
у сесії я їх не заміряв (це інтерактив, а не скриптовий API-виклик). Порівняння якості — реальне й
відтворюване з `predictions.csv`.

## Обмеження

1. Leakage завищує публічні метрики (ті самі ~0.98).
2. Без augmentation реалістичні приманки модель пропускає; augmentation покращує розпізнавання очевидних, але
   не тонких (~0.10).
3. `P(phish)` — проксі «читається як фішинг» (Spearman ρ≈0.49 на самих приманках), а не ground-truth.
4. Дані для augmentation синтетичні (є bias генератора) — для проду потрібні реальні корпоративні.
5. Лише текст: HTML-сигнали (mismatched href, форма-пароль) поки поза межами роботи.
6. Тренування на MPS (Intel Mac): batch 16 @ 256, 2 епохи (batch 32 призводить до OOM).

## Відтворення

Основний результат роботи — виконаний ноутбук з виводами; усі графіки й таблиці видно безпосередньо на GitHub.
Щоб ознайомитися з дослідженням, запускати нічого не потрібно. Нижче — інструкція для повторного прогону.

```bash
# 1. середовище (Python 3.12)
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 2. ноутбук самодостатній: F0 будує спліти з HuggingFace,
#    F2/F9 тренують (або завантажують model/ якщо є), F7 експортує ONNX.
export TOKENIZERS_PARALLELISM=false PYTORCH_ENABLE_MPS_FALLBACK=1 PYTHONPATH=src
jupyter nbconvert --to notebook --execute --inplace \
  --ExecutePreprocessor.timeout=5400 notebooks/phishing_scorer.ipynb
```

Ваги в git не зберігаються (256 MB × 2, щоб не заводити Git LFS). Вони або завантажуються з публічного W&B
(`scripts/fetch_models.py`, потрібен безкоштовний `wandb login`), або доступні через `--local`, або ноутбук
натреновує їх з нуля. Важкі файли (`model/`, `onnx/*.onnx`, спліти) додано в `.gitignore` — ноутбук їх
регенерує. F2/F9 ідемпотентні: якщо `model/…` присутні, повторний прогін не перетреновує; для перетренування
теку слід видалити.

Журнал спроб і всі порівняння прогонів — [`docs/EXPERIMENTS.md`](docs/EXPERIMENTS.md) та публічний W&B:
