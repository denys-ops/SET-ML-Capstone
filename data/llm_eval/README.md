# Порівняння з LLM (zero-shot)

Мета — чесно перевірити тезу «а чи не простіше було одразу взяти LLM», використавши реальний прогін, а не
числа з чужих статей.

## Склад

| Файл | Призначення |
|---|---|
| `sample_texts.csv` | 60 листів (30 phish / 30 legit, стратифіковано з `data/test.parquet`, seed 42) — лише текст, без міток |
| `_ground_truth.csv` | приховані істинні мітки + збережені `distilbert_p` / `baseline_p` для тих самих листів |
| `predictions.csv` | істина + вердикт LLM |
| `metrics.json` | F1/Acc/Prec/Rec трьох моделей |

## Методологія

Я витягнув тексти без міток; Claude Opus 4.8 класифікував кожен лист лише за текстом (мітки не бачив).
Після цього окремий скрипт звірив вердикти з істиною та з предикшенами DistilBERT (`p_phish ≥ 0.5`) і
baseline. Такий порядок гарантує, що LLM не має доступу до відповідей.

## Результат

Усі три моделі — F1 = 1.00, 0 розбіжностей. На перший погляд це виглядає дивно, але саме в цьому й суть:
на публічному тесті якість не є розрізняльним фактором (той самий leakage), розрізняють латентність
(10.5 мс на CPU проти сотень мс–секунд у API) і вартість (~$0 проти оплати за токени). Реактивний скоринг
на кожне натискання клавіші моя модель забезпечує, LLM — ні.

## Застереження щодо методу

- Одна LLM-сесія (Opus 4.8), n=60, лише текст — вибірка невелика.
- Латентність і вартість LLM — порядкові оцінки, у сесії я їх не заміряв (це інтерактив, а не скриптовий
  API-виклик). Тому цей крок не вбудовано в ноутбук, а винесено сюди окремими файлами для перевірки.
- Порівняння якості — реальне й відтворюване з `predictions.csv` + `metrics.json`.
- Реальне розходження в якості проявляється на корпоративному domain gap (див. головний README), а не на
  public test.
